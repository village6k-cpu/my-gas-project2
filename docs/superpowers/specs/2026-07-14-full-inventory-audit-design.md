# 빌리지 전체 재고 실사 모드 설계

## 문서 상태

- 작성일: 2026-07-14
- 상태: 사용자 승인 설계
- 구현 위치: `my-gas-project2/apps/today-dashboard`
- 사용자 접점: 헤이빌리 `재고` 탭
- 진실 원천: 승인 전에는 실사 세션, 승인 후에는 `village.equipment_ledger`

## 배경

빌리지는 최근 2~3년 동안 전체 재고 실사를 완료하지 못했다. 현재 재고 원장에는 275개 활성 품목과 장부상 1,073개 장비가 있으며, 현장 확인 완료 품목은 43개(16%)다. 분실·파손·수량 불일치를 줄이려면 먼저 전체 실사를 통해 총보유수량의 기준선을 확정해야 한다.

기존 헤이빌리 `재고` 탭의 `확인` 흐름은 장부 수량을 미리 보여주고, 직원이 입력한 실물 수량을 즉시 `equipment_ledger.stock_total`에 반영한다. 이 방식은 대여중·외부수리 장비가 있는 전수 실사에서 총보유수량을 잘못 줄일 수 있다.

이번 기능은 사장님이 모든 계수 과정에 개입하지 않아도 직원이 전체 매장을 실사하고, 완성된 결과만 사장님이 검토·승인할 수 있게 한다.

## 확정된 운영 조건

- 직원 1명이 약 3시간 동안 전체 매장을 실사할 수 있다. 이는 사장님의 실제 수행 경험에 근거한 처리량이다.
- 3시간 동안 반출·반납·장비 이동을 완전히 중지할 수 있다.
- 장비는 대체로 종류별로 정리되어 있으나 일부가 섞여 있다.
- 담당 직원은 장비명을 대체로 알지만 비슷한 모델·구성품에서 막힐 수 있다.
- 담당 직원은 로그인된 모바일 헤이빌리를 사용할 수 있다.
- 입력 항목이나 검수 품질을 줄이지 않는다.
- 오늘의 기본 할당 범위는 일부 품목이나 한 구역이 아니라 활성 품목 전체다.
- 직원의 모든 입력은 임시 저장하며, 사장님 승인 전에는 기존 원장을 변경하지 않는다.

## 목표

1. 직원 한 명이 모바일 헤이빌리에서 전체 매장의 상세 전수 실사를 수행한다.
2. 기존 장부 수량을 숨겨 기억이나 기대값에 맞춘 계수를 방지한다.
3. 정상·정비·파손 수량, 위치, 누락 구성품, 메모, 사진, 식별 확신도를 보존한다.
4. 대여중·정비중·식별불가 상태를 단순 분실과 구분한다.
5. 사장님은 완성된 세션을 한 번 검토하고 승인·수정·재실사를 결정한다.
6. 승인 전에는 `equipment_ledger`와 장비마스터 시트를 변경하지 않는다.
7. 승인 후에도 최초 관측값과 변경 전후 이력을 영구 보존한다.

## 비목표

- 첫 구현에서 모든 장비에 QR 또는 개별 시리얼 태그를 부착하지 않는다.
- 과거 2~3년의 모든 이동 이력을 재구성하지 않는다.
- `village-ai`에 새로운 웹 UI를 만들지 않는다.
- 실사 제출만으로 분실 확정, 폐기, 회계상 손실 처리를 자동 실행하지 않는다.
- 카카오톡이나 슬랙을 주 실사 입력 화면으로 사용하지 않는다.
- 이번 1차 구현에서 정기 순환실사 자동 배정까지 만들지 않는다. 다만 동일한 세션 모델을 이후 순환실사에 재사용할 수 있게 한다.

## 사용자 흐름

### 1. 실사 시작

헤이빌리 `재고` 탭 상단에 `전체 재고 실사 시작` 버튼을 추가한다.

시작 시 시스템은 다음을 수행한다.

- 담당자의 Supabase 사용자 ID와 이메일을 기록한다.
- 컷오프 시각을 기록한다.
- `전체 매장 장비 이동을 중지했습니다` 확인을 받는다.
- 활성 `equipment_ledger` 전 행을 실사 스냅샷으로 복제한다.
- 컷오프 시각의 활성 대여 거래와 스케줄 품목을 스냅샷으로 복제한다.
- 장비명·별칭 매칭 결과와 매칭 확신도를 함께 기록한다.
- 직원에게는 기존 총보유수량, 정비수량, 대여수량을 노출하지 않는다.

세션 생성과 스냅샷 작성은 `/api/inventory-audits/start`가 하나의 서버 작업으로 수행한다. 브라우저는 스냅샷 행을 직접 생성하거나 조회하지 않으며, 시작 응답에도 숨겨야 하는 수량을 포함하지 않는다.

매장 전체 전수 실사는 동시에 하나만 활성화한다. 기존 초안이 있으면 새 세션을 만들지 않고 이어서 연다.

### 2. 상세 계수

직원은 매장 동선대로 전체 장비를 확인한다. 검색과 대분류 필터는 탐색 보조일 뿐 입력 필드를 생략하지 않는다.

식별된 품목마다 다음을 기록한다.

- 장비 ID
- 보관 위치 또는 현재 구역
- 정상 수량
- 정비중 수량
- 파손·격리 수량
- 상태 미확인 수량
- 누락 구성품
- 특이사항
- 필요한 사진
- 식별 상태: `확실`, `식별 불확실`, `목록에 없음`

네 수량 버킷은 서로 배타적이며 합계가 매장에서 발견한 보유 실물 수량이다. 같은 품목을 다른 위치에서 다시 발견하면 기존 기록을 덮어쓰지 않고 위치별 관측을 추가한다. 검토 화면에서는 같은 장비 ID의 관측값을 합산한다.

식별이 불확실하거나 목록에 없는 장비는 임시 코드와 사진을 생성한다. 직원은 실사 중 사장님에게 물어보지 않고 해당 항목을 보류할 수 있다.

입력하지 않은 스냅샷 품목은 `0개`가 아니라 `미계수`다. `0개`는 직원이 해당 품목을 확인한 뒤 실물이 없다고 명시적으로 입력한 경우에만 성립한다.

### 3. 자동 저장과 제출

- 텍스트와 수량은 입력 즉시 서버와 IndexedDB 초안에 저장한다.
- 서버 저장이 실패하면 로컬 초안을 유지하고 재연결 시 같은 관측 ID로 재시도한다.
- 사진은 촬영 즉시 압축하고 전용 IndexedDB 업로드 대기열에 보존한다.
- 사진 업로드 실패는 수량 초안을 잃게 하지 않는다.
- 미전송 텍스트·사진이 있으면 제출을 막고 남은 건수를 표시한다.
- 제출 전까지 직원은 모든 관측을 수정할 수 있다.
- 제출 후 세션과 관측 원본은 잠기며 직원이 수정할 수 없다.

### 4. 사장님 검토

헤이빌리 `재고` 탭에 `검토할 실사` 카드를 표시한다. 검토 화면은 다음 묶음을 제공한다.

- 실물 + 확실히 연결된 대여중 수량이 기존 장부와 일치
- 수량 차이
- 파손·정비·구성품 문제
- 식별 불확실
- 목록에 없는 신규 장비
- 미계수
- 대여 장비명 연결이 불확실
- 실사 시작 후 원장이 바뀐 충돌 항목

사장님은 항목별로 다음 결정을 저장한다.

- 실사 결과 승인
- 최종 수량·상태를 수정한 뒤 승인
- 기존 장부 유지
- 재실사 요청

문제가 없는 일치 항목은 목록을 검토한 뒤 일괄 승인할 수 있다. 그러나 일치 항목도 승인 전에는 원장에 반영하지 않는다.

### 5. 승인 적용

모든 필수 항목에 결정이 있으면 세션을 최종 승인할 수 있다.

최종 승인은 하나의 서버 전용 트랜잭션으로 다음을 수행한다.

1. 세션과 결정 행을 잠근다.
2. 승인자가 사장님 계정인지 다시 확인한다.
3. 스냅샷 이후 원장 변경 충돌이 없는지 확인한다.
4. 승인된 최종 총보유수량, 정비수량, 상태, 문제 목록을 `equipment_ledger`에 반영한다.
5. 변경된 품목마다 `equipment_events`에 실사 세션 ID, 변경 전후 값, 계수자, 승인자, 시각을 기록한다.
6. 실사 결과를 적용한 품목은 마지막 관측 시각과 계수자를 `last_verified_at`, `last_verified_by`에 기록한다.
7. 승인된 문제나 정비 상태가 있으면 `verify_status=attention`, 없으면 `verify_status=verified`로 기록한다.
8. 세션을 `approved`로 전환한다.

중간에 하나라도 실패하면 전체 트랜잭션을 롤백한다. 승인된 실사 원본과 사진은 이후에도 삭제하지 않는다.

Supabase 승인이 끝난 뒤 장비마스터 미러링 작업을 별도로 실행한다. Supabase 원장이 승인 후 진실 원천이며, Google Sheets 미러링 실패는 승인 원장을 되돌리지 않는다. 대신 세션에 `mirror_pending` 또는 `mirror_failed` 상태를 표시하고 재시도한다.

## 정보 구조와 컴포넌트

기존 `InventoryView.tsx`는 이미 재고 목록, 검색, 장비 추가, 문제 편집, 블랙박스 등 여러 책임을 가진 큰 컴포넌트다. 실사 로직을 여기에 직접 누적하지 않는다.

- `InventoryView`: 실사 시작·이어하기·검토 카드만 노출한다.
- `InventoryAuditView`: 직원용 전체 실사 화면과 세션 복구를 담당한다.
- `InventoryAuditObservationForm`: 위치별 수량 버킷, 구성품, 메모, 식별 상태를 담당한다.
- `InventoryAuditReview`: 사장님 비교·결정·일괄 승인 화면을 담당한다.
- `lib/inventory-audit/aggregation.ts`: 위치별 관측 합산과 분류를 담당하는 순수 함수다.
- `lib/inventory-audit/reconciliation.ts`: 스냅샷, 대여중 수량, 관측값을 비교하는 순수 함수다.
- `lib/inventory-audit/offline.ts`: 초안과 재전송 대기열을 담당한다.
- `/api/inventory-audits/start`: 직원 인증 후 세션과 숨김 스냅샷을 생성한다.
- `/api/inventory-audits/*`: 사장님 전용 검토·최종 승인과 증거 URL 발급을 담당한다.

직원 초안 저장은 Supabase RLS를 사용하고, 최종 승인은 브라우저에서 직접 `equipment_ledger`를 수정하지 않고 서버 API만 통과한다.

## 데이터 모델

### `village.inventory_audit_sessions`

- `id uuid primary key`
- `mode text`: 1차 구현은 `full_shop`
- `status text`: `draft`, `submitted`, `in_review`, `recount_requested`, `approved`, `cancelled`
- `cutoff_at timestamptz`
- `started_by uuid`, `started_by_email text`
- `movement_frozen boolean`
- `started_at`, `submitted_at`, `approved_at`
- `approved_by uuid`, `approved_by_email text`
- `parent_session_id uuid`: 재실사 연결
- `mirror_status text`: `not_ready`, `pending`, `synced`, `failed`
- `created_at`, `updated_at`

### `village.inventory_audit_snapshot_items`

세션 시작 시 품목별 장부와 대여 상태를 고정한다.

- `session_id uuid`
- `equipment_id text`
- 장비명, 별칭, 대분류, 카테고리
- `ledger_stock_total`, `ledger_stock_maint`, `ledger_state`, `ledger_open_issues`
- `ledger_updated_at`
- `active_rental_qty`
- `active_rental_refs jsonb`
- `rental_match_status text`: `matched`, `ambiguous`, `unmatched`, `none`
- 기본키: `(session_id, equipment_id)`

대여 스냅샷은 취소·반납완료 거래, 세트 헤더, 제외 품목, 매장 현장 제공 품목을 제외한다. 장비명과 원장 이름·별칭의 연결이 확실한 경우만 `active_rental_qty`에 포함한다. 애매한 매칭은 자동 합계에 넣지 않고 검토 예외로 남긴다.

컷오프 시각에 대여 기간이 겹치더라도 실제 반출 근거가 없는 예약은 확정 대여수량으로 취급하지 않는다. `taken_qty > 0` 또는 `checkout_state=taken`인 스케줄 품목만 확정 수량으로 계산하고, 필드가 서로 모순되거나 누락된 경우에는 `ambiguous`로 분류한다.

### `village.inventory_audit_observations`

하나의 장비가 여러 위치에서 발견될 수 있으므로 품목당 한 행이 아니라 위치별 관측 행을 저장한다.

- `id uuid primary key`: 클라이언트가 생성해 재시도를 멱등 처리
- `session_id uuid`
- `equipment_id text null`
- `temporary_code text null`
- `temporary_label text null`
- `location text`
- `count_normal int`
- `count_maintenance int`
- `count_damaged int`
- `count_condition_unknown int`
- `missing_components jsonb`
- `note text`
- `identification_status text`: `confirmed`, `uncertain`, `unlisted`
- `evidence_refs jsonb`
- `observed_by uuid`, `observed_by_email text`
- `client_updated_at`, `created_at`, `updated_at`

모든 수량은 0 이상의 정수다. 식별 상태가 `confirmed`면 `equipment_id`가 필수이고, 그 외에는 임시 코드가 필수다.

### `village.inventory_audit_decisions`

- `id uuid primary key`
- `session_id uuid`
- `equipment_id text null`: 기존 원장 품목 결정
- `source_observation_id uuid null`: 식별 불확실·목록 외 관측 결정
- `decision text`: `apply_audit`, `keep_ledger`, `recount`
- `resolution text`: `existing_equipment`, `create_equipment`, `not_inventory`
- `resolved_equipment_id text null`: 기존 품목 연결 또는 신규 생성 후 확정 ID
- `new_equipment_payload jsonb null`: 신규 생성 시 장비명, 대분류, 카테고리, 단가, 최초 상태
- `final_stock_total int null`
- `final_stock_maint int null`
- `final_state text null`
- `final_open_issues jsonb`
- `other_confirmed_offsite_qty int`
- `review_note text`
- `reviewed_by uuid`, `reviewed_by_email text`, `reviewed_at`
- 같은 세션에서 `equipment_id` 또는 `source_observation_id` 중 정확히 하나만 존재해야 한다.
- 기존 품목 결정은 `(session_id, equipment_id)`, 임시 관측 결정은 `(session_id, source_observation_id)`가 각각 유일하다.

식별 불확실·목록 외 장비는 기존 장비 연결, 신규 원장 품목 생성, 재고 아님 중 하나를 결정하기 전까지 세션 최종 승인을 막는다. 신규 품목 생성과 해당 실사 수량 적용도 같은 최종 승인 트랜잭션에서 처리한다.

## 수량 계산과 비교 규칙

품목별 매장 실물 합계는 다음과 같다.

`physical_total = normal + maintenance + damaged + condition_unknown`

검토용 후보 총보유수량은 다음과 같다.

`candidate_total = physical_total + matched_active_rental_qty + owner_confirmed_other_offsite_qty`

이 값은 비교 제안일 뿐 자동 반영값이 아니다. 사장님이 `apply_audit` 결정을 저장할 때 최종 총보유수량을 확정한다.

- 매장에서 직접 센 정비중 장비는 `physical_total`에 이미 포함되므로 다시 더하지 않는다.
- 외부 수리·직원 보관 등 매장 밖 장비는 사장님이 근거를 확인한 뒤 `other_confirmed_offsite_qty`로 추가한다.
- 식별이 불확실한 대여 품목은 후보 총보유수량에 더하지 않는다.
- 미계수 품목은 후보 총보유수량을 만들지 않는다.
- 관측 0은 명시적 0이며 미계수와 다르다.

## 권한과 보안

- 로그인 직원은 자신이 시작한 `draft` 세션과 관측을 생성·조회·수정할 수 있다.
- 로그인 직원은 세션을 제출할 수 있지만 제출 후에는 수정할 수 없다.
- 다른 직원의 초안은 직접 수정할 수 없다.
- 스냅샷 테이블은 직원 클라이언트가 직접 조회할 수 없고 서버 검토 API만 읽는다.
- 사장님 여부는 서버에서 Supabase `getUser()`로 검증한 이메일과 `INVENTORY_OWNER_EMAILS` 환경변수를 비교한다.
- 사장님 검토·승인 API는 서비스 역할 키를 서버에서만 사용한다.
- 서비스 역할 키는 클라이언트 번들에 포함하지 않는다.
- 승인 SQL 함수는 브라우저 역할과 `PUBLIC`의 실행 권한을 제거하고 서비스 역할만 호출할 수 있게 한다.
- 감사 테이블과 증거 저장소에는 RLS를 적용한다.
- 감사 사진은 비공개 Supabase Storage 버킷 `inventory-audit-evidence`에 저장한다.
- 객체 경로는 `{session_id}/{observation_id}/{uuid}.jpg`를 사용하고 DB에는 경로와 메타데이터만 저장한다.
- 직원은 자신의 초안 세션 경로에만 업로드할 수 있다.
- 검토 화면은 짧은 만료시간의 서명 URL로 사진을 조회한다.
- `full_shop` 세션이 `draft`인 동안에는 `equipment_ledger`에 대한 일반 authenticated UPDATE/INSERT/DELETE를 RLS에서 거부한다. 서비스 역할의 승인 트랜잭션만 이 잠금을 우회한다.
- 세션 제출 후에는 매장 운영을 재개할 수 있도록 일반 원장 쓰기 잠금을 해제한다. 제출 이후 발생한 원장 변경은 승인 시 버전 충돌로 잡는다.

## 오프라인과 오류 처리

- 관측 ID를 클라이언트에서 생성해 재시도 시 중복 관측을 막는다.
- 로컬 초안과 서버 응답에 수정 시각을 기록하고, 같은 관측의 충돌은 최신 시각으로 조용히 덮지 않고 사용자에게 선택을 요구한다.
- 앱 재시작 시 활성 초안과 전송 대기열을 복구한다.
- 사진은 최대 크기로 압축하고 5회 지수 백오프로 재시도한다.
- 미전송 항목이 있으면 제출을 차단한다.
- 제출 API는 현재 상태가 `draft`일 때만 `submitted`로 바꾸며 반복 요청에는 같은 결과를 반환한다.
- `draft` 상태의 `full_shop` 세션이 있으면 기존 즉시 원장 수정 `확인` 버튼을 비활성화하고, 같은 조건을 RLS에서도 강제한다.
- 승인 시 현재 `equipment_ledger.updated_at`과 스냅샷 값을 비교한다. 바뀐 품목은 충돌로 표시하고 결정 재확인 전까지 최종 승인을 막는다.
- 장비마스터 미러링 실패는 재시도 가능 상태로 남기고 사장님 화면에 표시한다.

## 상태 전이

```text
draft -> submitted -> in_review -> approved
                    -> recount_requested -> 새 draft(parent_session_id 연결)
draft -> cancelled
```

사장님이 제출 세션을 처음 열면 `submitted`에서 `in_review`로 전환한다. `submitted` 이후의 원본 관측은 불변이다. 검토 중 수정은 관측값을 고치는 대신 별도 결정 행에 기록한다.

## 테스트 전략

### 순수 로직 테스트

- 위치별 관측 합산
- 네 수량 버킷 합계
- 미계수와 명시적 0 구분
- 대여중 확정 매칭만 후보 총수량에 포함
- 정비중 장비 이중 합산 방지
- 불확실·목록 외·미계수 분류
- 장부 변경 충돌 탐지

### 데이터·권한 테스트

- 직원은 자신의 초안만 수정 가능
- 직원은 제출된 관측을 수정할 수 없고, `draft` 전체 실사 중에는 일반 원장 쓰기도 불가
- 직원은 숨김 스냅샷 수량을 직접 조회 불가
- `draft` 전체 실사 중 일반 원장 쓰기가 RLS에서 거부됨
- 사장님 API가 아닌 호출은 승인 거부
- 서비스 역할 전용 승인 함수의 공개 실행 권한 없음
- 최종 승인 중 한 항목 실패 시 원장 전체 롤백
- 승인 성공 시 품목별 `equipment_events` 생성
- 비공개 사진 버킷의 세션 소유권 정책

### 화면 테스트

- 직원 실사 화면에 기존 수량·대여수량이 노출되지 않음
- 활성 275개 품목을 한 세션에서 다룰 수 있음
- 새로고침·오프라인 후 이어하기
- 같은 품목의 여러 위치 관측
- 사진 실패와 재시도 표시
- 미전송 작업이 있을 때 제출 차단
- 제출 후 편집 잠금
- 사장님 일치 항목 일괄 결정과 예외 항목 개별 결정
- 진행 중 실사에서 기존 `확인` 비활성화

### 프로덕션 스모크 테스트

1. 배포 직전 `equipment_ledger` 행 수와 주요 집계를 기록한다.
2. 테스트용 빈 실사 세션을 생성하고 취소한다.
3. 원장 행 수와 집계가 전혀 변하지 않았음을 확인한다.
4. 실제 로그인 직원 계정에서 모바일 시작·이어하기·취소를 확인한다.
5. 사장님 계정과 직원 계정의 승인 권한 차이를 확인한다.
6. 쓰기 테스트는 감사 전용 테이블과 비공개 사진 버킷에만 수행한다.

## 직원 안내문

1. 장비 이동이 중지됐는지 확인합니다.
2. 헤이빌리 `재고 > 전체 재고 실사 시작`을 누릅니다.
3. 매장 동선대로 모든 장비의 수량·위치·상태·구성품을 기록합니다.
4. 모델이 애매하면 사장님께 묻지 말고 사진을 찍어 `미확인`으로 남깁니다.
5. 파손·누락·정비 장비는 상태와 사진을 남깁니다.
6. 전체 매장을 다 돈 뒤 미전송 항목이 0건인지 확인하고 제출합니다.
7. 제출만으로 기존 재고 원장은 바뀌지 않습니다.

## 완료 기준

- 직원 한 명이 전체 활성 품목을 대상으로 상세 전수 실사 세션을 진행할 수 있다.
- 입력 품질이나 필수 항목을 줄이는 품목 수 제한이 없다.
- 기존 장부 수량은 직원 화면에서 숨겨진다.
- 수량·위치·상태·구성품·메모·사진·식별 확신도가 보존된다.
- 미계수와 명시적 0이 데이터와 화면에서 구분된다.
- 오프라인·새로고침 후 작업을 이어갈 수 있다.
- 제출 전에는 수정 가능하고 제출 후 원본은 불변이다.
- 제출만으로 `equipment_ledger`와 Google Sheets가 변경되지 않는다.
- 사장님 승인 후에만 원장이 트랜잭션으로 변경된다.
- 변경 전후 값, 계수자, 승인자, 시각, 세션 ID가 감사 이력에 남는다.
- 시트 미러링 실패가 승인된 Supabase 원장을 되돌리지 않고 재시도 상태로 남는다.
